# HTTP协议

# 1、状态码类别（背下来）

  • 1XX Informational（信息性状态码） 接受的请求正在处理（服务器收到请求）
  • 2XX Success（成功状态码） 请求正常处理完毕（请求成功，服务器说：可以没问题，给你吧，如00）
  • 3XX Redirection（重定向状态码） 需要进行附加操作以完成请求（服务器说：你到我这里来不行，你去另一个地方去，然后浏览器帮给你另一个地址，如302）
  • 4XX Client Error（==客户端==错误状态码） 服务器无法处理请求（比如客户端请求了一个根本不存在的地址）（客户端的锅）
  • 5XX Server Error（==服务器==错误状态码） 服务器处理请求出错（服务器的锅）

# 2、http 常见的状态码有哪些？

  • 200 成功
  • 301 永久重定向 （配合location，浏览器自动处理）
    • 一个网站域名到期或者主人想更换域名了，旧的域名就可以返回一个301的状态码然后返回一个location新的域名，浏览器就记住了这件事情，下次就不会再去访问旧的域名
  • 302 临时重定向 （配合location，浏览器自动处理）
    • 区别是下次浏览器仍然会再去访问原来的地址（只是这次请求会访问新的地址）
    • 如百度搜索Javascript，出来的结果有一个菜鸟教程JavaScript，但是打开F2发现它的网址是百度的，而不是菜鸟教程的网址，但是点开这个搜索结果仍任是去到了菜鸟教程，这就是因为做了临时重定向
  • 304 资源违未被修改
    • 服务端说你请求的资源已经请求过了而且在本地还有效，你不用再重新请求一遍了，我也不会再给你发一遍了，你直接用原来的就行。
  • 404 资源未找到
    • 服务端说我这没有你要的东西
  • 403 没有权限
    • 服务端说你丫还没登录呢就想获取资源
  • 500 服务器错误
    • 这时最常见的服务器错误，后端的代码有bug了，直接宕机，找后端同事解决
  • 504 网关超时
    • 能访问通服务器，但是服务器也不是一台机器，有可能是很多台机器，服务器们内部连接跳转时可能会超时
  • 关于协议和规范
    • 就是一个约定
    • 要求大家都跟着执行
    • 不要违反规范，例如 早期的 IE 浏览器（最终被淘汰）

# 3、method

  • 传统的method
    • get 获取服务器的数据
    • post 向服务器提交数据
    • 简单的网页功能，就这两个操作
  • 现在的method
    • get 获取数据
    • post 不是获取数据了，而是**新建数据（**比如创建一篇博客/留言）
    • path/put 更新数据（比如更新一篇博客/留言）
    • delete 删除数据

# 5、http headers 请求头（背）

  • 常见 Request Headers（客户端往服务端发的）
    • Accept 浏览器可接接收的数据格式
    • Accept-Encoding 浏览器可接受的压缩算法，如 gzip
    • Accept-Language 浏览器可接收的语言，如 zh-CN
    • Connection: keep-alive 一次TCP连接重复使用
    • cookie
    • Host 请求的域名
    • User-Agent（简称UA） 浏览器信息（甚至里面包含各种电脑系统、手机系统的信息）
    • Content-type 发送数据的格式，如 application / JSON
  • 常见 Response Headers（服务端向客户端返回的）
    • Content-type 返回的数据的格式，如 application / JSON
    • Content-length 返回数据的大小，多少字节
    • Content-Encoding 返回数据的压缩算法，如 gzip
    • Set-cookie 服务端改cookie时通过Set-cookie来改
  • 缓存相关的headers
    • Cache-Control （Response header 中）
    • Last-Modified
    • Etag
  • 额外知识点：可以自定义 header

# 6、缓存

  • 什么是缓存？
    • 浏览器访问一个新的网站，服务器会把所有的数据原原本本给你，但是第二次访问时，就没有再把所有资源都访问一变的必要。缓存就是把一些没有必要重新获取一遍的东西，存一份，后续就不再去获取。
  • 为什么需要缓存？
    • 让页面加载得更快一些。整个过程 中，网络请求是最耗时的一个环节， 网络请求的加载相比于CPU的计算包括页面的渲染都是很慢的，因此我们要解决这个问题。要尽量减小网络请求的体积和数量，这样才能让网络请求更快一些。
  • 哪些资源可以被缓存？
  • 静态资源。如 js、css、img，注意html是不能被缓存的，因为它随时会被更新上线

# 7、强制缓存

  • Cache-Control

  • Response Headers中

  • 控制强制缓存的逻辑

  • 例如 Cache-Control: max-age=31536000（单位是秒, 换算得到 1 年)

  • 客户端再次发送请求

    • 之前已经有了Cache-Control，客户端会把资源缓存下来，再次请求时只要判断Cache-Control的时间如果还有效，就不会请求服务器，而是读取本地缓存。这一次就没有经过网络，而是直接读取本地缓存，比请求网络快了很多。

  • 缓存时间有可能是设置半年或一年，总有一天缓存会失效。缓存失效的情况下就会再次请求服务端，服务端会重新返回这个资源和 Cache-Control，又设置一遍缓存时间。如此循环。

  • Cache-Control 的值：

  • max-age 缓存时长

  • no-cache 不使用本地缓存

  • no-store 不使用本地缓存而且也不使用服务端的一些缓存措施（不常见）

  • private 只允许用户做缓存

  • public 允许中间的路由、代理做缓存

# 8、协商缓存

  • 是服务端缓存策略（服务端来判断一个资源是否可以被缓存）
    • 注意：不要理解为这个资源缓存在服务端哈哈哈，缓存缓存，指的是不用网络就能获取的，缓存在服务端我请求还要通过网络，那还缓存个P！
  • 服务器判断客户端资源是否和服务端一致
  • 一致则返回 304（ 资源违未被修改），否则返回200和最新的资源

###9、资源标识

  • 在Response Header 中，有两种

    • Last-Modified 资源的最后修改时间
    • Etag 资源的唯一标识 （一个字符串，类似人类的指纹）
    • 以上两者如果共存，会优先使用Etag
    • 因为Last-Modified只能精确到秒级
    • 如果资源被重复生成，而内容不变，则Etag个精确

# 11、三种刷新操作

  • 正常操作：地址栏输入 url 跳转链接，前进后退等

    • 强制缓存有效，协商缓存有效

  • 手动刷新：F5，或者点击刷新按钮，或者右击菜单刷新

    • 强制缓存失效，协商缓存有效

  • 强制刷新： ctrl + F5

    • 强制缓存失效，协商缓存失效
